The core problem I chose to solve is the massive gap between raw enterprise data and actionable business insights. In traditional organizations, business stakeholders have messy, massive data dumps spanning millions of rows in Excel or CSV. To extract any value or dashboards from that data, they either have to learn complex Python Pandas scripting themselves—which is an incredibly steep learning curve—or log a ticket with a Data Analyst team and wait weeks.

The Python code executes in the sandbox, computes the ground-truth mathematical dataframe, and then the backend FastAPI server aggressively intercepts the HTML payload and natively injects the JSON tree directly into the JavaScript context just milliseconds before streaming it to the React frontend. The result is mathematically flawless charts seamlessly rendered in an iframe sandbox. SECTION 3: TRADEOFFS (3-4 MINUTES)
----------------------------------------------------------------------------------------
👉 VISUAL: Switch back to the BI Tool UI chat interface, perhaps toggling dark / light mode to show the new theme capabilities.

SPEAKER:
"In building this, I made a few heavy technical tradeoffs. 

The first tradeoff was Session Storage. Instead of standing up a heavy Postgres or Redis cluster, I built a custom Thread-Safe LRU memory store in the backend (`session_store.py`). To prevent memory exhaustion during heavy EDA, I implemented an intelligent paging system: when memory pressure spikes, inactive Pandas Dataframes are compressed and paged tightly to the local disk, keeping the read/write speed blisteringly fast when a user queries it, but keeping infrastructure complexity down. If I were scaling this horizontally, I would trade this off for a formal Redis cluster, but for single-tenant or team deployment, the latency wins here are massive.

The second massive tradeoff was using the `multiprocessing` library for the Sandbox instead of spinning up isolated Docker containers for every query. A Docker container is vastly more secure, but provisioning a cold-start container takes anywhere from 400 to 800 milliseconds. For an interactive chat application natively supporting Voice interaction, adding nearly a second of latency on top of LLM generation ruins the user experience. By trading off raw containerization for a rigorously defined AST-parser and a strict multiprocessing boundary, I kept the response execution time under 50 milliseconds while maintaining very robust isolation.

The final tradeoff was instructing the AI to build raw HTML artifacts natively, rather than having the AI return raw JSON datasets that my React frontend would map to predefined components like Recharts or D3. By letting the AI write raw HTML/JS, I traded away strict UI conformity in exchange for infinite flexibility. The AI can natively import Chart.js, build completely custom grid layouts, or write whatever bespoke KPI card visualizations fit the data best natively."

SECTION 4: FAILURE MODES (2 MINUTES)
----------------------------------------------------------------------------------------
👉 VISUAL: Open `backend/services/code_executor.py` again and highlight lines involving `proc.join(timeout=CODE_EXECUTION_TIMEOUT)` and `proc.terminate()`.

SPEAKER:
"Finally, let's talk about what happens when the system breaks, and how I engineered it to fail gracefully.

The most catastrophic failure mode is an infinite loop or a memory-crashing Cartesian explosion—for instance, if the AI tells pandas to merge two massive dataframes on identical non-unique keys without limits. To defend against this, the isolated worker process is tethered to a strict watchdog timer. If the OS-level process does not fully complete mathematical evaluation and serialization within a 30-second window, the main thread brutally sends a SIGKILL command using `proc.terminate()`, instantly reclaiming the memory and returning a graceful 'Execution Timeout' error to the UI without touching the main web server.

Another common failure mode is hallucinated column references. If the user asks a confusing question, the AI might generate syntax-perfect Pandas code that attempts to index a column named `'Sales'`, but the true schema column is named `'sales_revenue'`. In this scenario, the Python sandbox catches the exact `KeyError` traceback, serializes the exception, and pipes the failure back to the frontend. The system is designed to eventually parse these errors so the agent can silently correct itself, but currently, it safely outputs the execution traceback so the user knows exactly why the visualization failed to build.
